Endpoints para gestionar dominios de envío: registro, verificación DNS, estado y perfil de remitente.

**Base URL:** `https://api.reallyquickemails.com`

---

## Autenticacion

Todos los endpoints de dominios requieren Secret Key:

| Header          | Tipo   | Requerido | Descripción                       |
| --------------- | ------ | --------- | --------------------------------- |
| `Authorization` | string | Sí        | `Bearer sk_proj_...`              |
| `Content-Type`  | string | Sí*       | `application/json` (en POST/PUT). |

---

## Flujo de integracion

Si automatizas el alta de dominios desde tu propia aplicación, este es el recorrido completo:

<Steps>
  <Step title="Registra el dominio">
    `POST /domains/register` retorna los 7 registros DNS en `dns_records`, ya listos para mostrar.
  </Step>
  <Step title="Ofrece la configuración en un clic">
    `GET /domains/{domain}/domain-connect`. Si retorna `supported: true`, redirige al usuario a `apply_url` y su proveedor crea los registros solo. Si no, muestra los registros para configuración manual.
  </Step>
  <Step title="Verifica">
    `POST /domains/{domain}/verify` consulta el estado real y actualiza los registros. Llámalo cuando el usuario indique que ya configuró su DNS.
  </Step>
  <Step title="Confirma antes de enviar">
    Recibe `domain.verified` por webhook, o consulta `GET /domains/{domain}/status`. El dominio puede enviar cuando `can_send: true`.
  </Step>
</Steps>

<Tip>
  **No pollees: escucha el webhook.** Cuando un dominio queda verificado, RQE emite `domain.verified` hacia la URL de webhooks de tu proyecto (y `domain.failed` si la verificación falla). Se emite **una sola vez**, en la transición. Ver [Webhooks](/api-reference/webhooks#domainverified-domainfailed).
</Tip>

<Info>
  Los dominios en `pending` se re-verifican solos, sin que llames a nada: con frecuencia alta el primer día y espaciándose a medida que el alta envejece. Si tu usuario publica el DNS días después del registro, igual recibirás el webhook. `POST /verify` sigue siendo útil para respuesta inmediata cuando el usuario dice "ya lo configuré".
</Info>

<Note>
  **SPF y DMARC no se validan automáticamente.** Siempre aparecen con `status: "not_set"` en `dns_records`, incluso si están correctamente publicados. `can_send` no los considera. Si necesitas comprobar que tu usuario los configuró, resuélvelos por tu cuenta con una consulta DNS.
</Note>

## POST /domains/register

Registra un dominio de envío: crea la identidad de envío, genera los registros DNS y crea el perfil de remitente.

### Request Body

| Campo          | Tipo   | Requerido | Descripción                                                        |
| -------------- | ------ | --------- | ------------------------------------------------------------------ |
| `domain`       | string | Sí        | Dominio a registrar (ej. `mitienda.com`).                          |
| `sender_name`  | string | Sí        | Nombre del remitente (ej. `Mi Tienda`).                            |
| `sender_email` | string | Sí        | Dirección de correo del remitente. Debe terminar en `@{domain}` (ej. `ventas@mitienda.com`). |
| `external_key` | string | No        | Clave externa para integraciones de terceros.                      |

### Ejemplo

<Tabs>

<Tab title="cURL">

```bash
curl -X POST https://api.reallyquickemails.com/domains/register \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer sk_proj_xxxxxxxxxxxx" \
  -d '{
    "domain": "mitienda.com",
    "sender_name": "Mi Tienda",
    "sender_email": "ventas@mitienda.com"
  }'
```

</Tabs>

### Registros DNS generados

| # | Tipo | Nombre | Propósito | Notas |
|---|------|--------|-----------|-------|
| 1 | TXT | `_amazonses` | Verificación de dominio | Valor generado automáticamente al registrar |
| 2-4 | CNAME | `{token}._domainkey` | DKIM | 3 registros, tokens generados automáticamente |
| 5 | TXT | `@` (raíz del dominio) | SPF | `v=spf1 include:amazonses.com ~all` |
| 6 | TXT | `_dmarc` | DMARC | `v=DMARC1; p=quarantine; rua=mailto:dmarc-reports@{domain}` |
| 7 | CNAME | `bounce` | Return-Path | Apunta a `feedback-smtp.us-east-1.amazonses.com` |

> **Nota:** Los nombres son relativos al dominio. Con dominio `mitienda.com`, el registro `_amazonses` se configura como `_amazonses.mitienda.com`. Algunos proveedores agregan el dominio automáticamente, así que solo ingresas `_amazonses`.

Configura los registros DNS en tu proveedor y llama a `POST /domains/{domain}/verify` para validar. Ver [Deliverability](/concepts/deliverability).

### Codigos de Error

| Código | Descripción                                              |
| ------ | -------------------------------------------------------- |
| `400`  | Campos requeridos faltantes, dominio con formato inválido, o `sender_email` no termina en `@{domain}`. |
| `401`  | API Key inválida.                                        |
| `409`  | El dominio ya está registrado en este proyecto.          |
| `500`  | Error interno al registrar el dominio.                   |


La verificación comprueba tres aspectos:

1. **Verificación de dominio** — El registro TXT `_amazonses` está configurado.
2. **DKIM** — Los 3 registros CNAME de DKIM están configurados y propagados.
3. **MAIL FROM** — El registro CNAME de Return-Path (`bounce`) está configurado.

Campos clave:

| Campo | Descripción |
|---|---|
| `verification_status` | `verified` (dominio y DKIM verificados), `failed` (alguna verificación falló) o `pending`. |
| `details.domain_verification` | Estado crudo de la verificación de dominio: `Success`, `Pending`, `Failed`, `TemporaryFailure` o `NotStarted`. |
| `details.dkim_verification` | Estado crudo de DKIM (mismos valores). |
| `details.mail_from_status` | Estado crudo de MAIL FROM (mismos valores). |
| `sender_profile` | Perfil de remitente activo del dominio, o `null` si no existe. |
| `can_send` | `true` cuando dominio y DKIM están verificados. |

`can_send` es `true` cuando `details.domain_verification` y `details.dkim_verification` son `Success` (equivalente a `verification_status: "verified"`). MAIL FROM no es obligatorio para enviar, pero mejora la entregabilidad.

| Código | Descripción                                       |
| ------ | ------------------------------------------------- |
| `401`  | API Key inválida.                                 |
| `404`  | Dominio no encontrado en el proyecto.             |
| `500`  | Error al comunicarse con la infraestructura de envío. |

| `limit`   | integer | No        | Resultados por página. Default `50`, máximo `200`.                                |

## GET /domains/:domain/domain-connect

Configuración de DNS en un clic. Si el proveedor DNS del dominio soporta el estándar [Domain Connect](https://www.domainconnect.org/), retorna una URL firmada que crea los 7 registros automáticamente: el usuario final solo confirma en la pantalla de su proveedor, sin copiar nada a mano.

Úsalo justo después de `POST /domains/register`. Si el proveedor no lo soporta, cae de vuelta a mostrar los registros de `dns_records` para configuración manual.

<Info>
  Cobertura medida sobre dominios reales: alrededor del 50% soporta Domain Connect. Cloudflare está operativo. Siempre debes manejar el caso `supported: false`.
</Info>

### Parametros de Ruta

| Parámetro | Tipo   | Descripción                              |
| --------- | ------ | ---------------------------------------- |
| `domain`  | string | Dominio registrado (ej. `mitienda.com`). |

### Query Parameters

| Parámetro      | Tipo   | Requerido | Descripción                                                        |
| -------------- | ------ | --------- | ------------------------------------------------------------------ |
| `redirect_uri` | string | No        | URL a la que vuelve el usuario tras aplicar los registros. Ver nota. |

<Warning>
  `redirect_uri` solo acepta hosts de ReallyQuickEmails (`app.reallyquickemails.com`) y de desarrollo local. Si envías el dominio de tu propia aplicación, la respuesta es `supported: false` con `reason: "invalid_redirect_uri"`. Omite el parámetro: la `apply_url` sigue siendo válida, solo que el usuario termina en la página de su proveedor DNS en vez de volver a tu aplicación.
</Warning>

### Ejemplo

<Tabs>

<Tab title="cURL">

```bash
curl -X GET "https://api.reallyquickemails.com/domains/mitienda.com/domain-connect" \
  -H "Authorization: Bearer sk_proj_xxxxxxxxxxxx"
```

</Tab>

</Tabs>

**Respuesta** `200 OK` — proveedor compatible

```json
{
  "supported": true,
  "provider_name": "Cloudflare",
  "zone": "mitienda.com",
  "host": "",
  "apply_url": "https://dash.cloudflare.com/domainconnect/v2/domainTemplates/providers/reallyquickemails.com/services/email-sending/apply?domain=mitienda.com&sesVerify=...&sig=...&key=_dck1"
}
```

Redirige al usuario a `apply_url`. Al confirmar, su proveedor crea los registros. Después llama a `POST /domains/{domain}/verify` para confirmar el estado.

**Respuesta** `200 OK` — proveedor no compatible

```json
{
  "supported": false,
  "reason": "no_domain_connect",
  "zone": "mitienda.com",
  "host": ""
}
```

### Campos de respuesta

| Campo           | Descripción                                                                       |
| --------------- | --------------------------------------------------------------------------------- |
| `supported`     | `true` si hay un flujo de un clic disponible para este dominio.                    |
| `apply_url`     | URL firmada a la que debes redirigir. Solo presente cuando `supported: true`.      |
| `provider_name` | Nombre del proveedor DNS detectado (ej. `Cloudflare`, `GoDaddy`).                  |
| `zone`          | Zona DNS detectada. Para subdominios, el dominio raíz que tiene los NS.            |
| `host`          | Prefijo del subdominio dentro de la zona. Cadena vacía si el dominio es la raíz.   |
| `reason`        | Motivo por el que no hay flujo disponible. Solo presente cuando `supported: false`. |

### Valores de `reason`

| Valor                      | Significado                                                                  |
| -------------------------- | ---------------------------------------------------------------------------- |
| `no_zone`                  | No se pudo determinar la zona DNS del dominio.                               |
| `no_domain_connect`        | El proveedor DNS no publica soporte para Domain Connect.                      |
| `no_sync_flow`             | El proveedor soporta Domain Connect pero no el flujo interactivo.            |
| `template_not_onboarded`   | El proveedor no tiene habilitada la plantilla de ReallyQuickEmails.          |
| `identity_missing_tokens`  | El dominio aún no tiene tokens generados. Registra el dominio primero.       |
| `invalid_redirect_uri`     | El `redirect_uri` enviado no está permitido.                                  |
| `signing_key_not_configured` | Error de configuración del servicio. Contacta a soporte.                    |

<Note>
  `supported: false` no es un error: la respuesta es `200 OK`. Muestra los registros DNS de `GET /domains/{domain}/dns-records` para que el usuario los configure manualmente.
</Note>

### Codigos de Error

| Código | Descripción                           |
| ------ | ------------------------------------- |
| `401`  | API Key inválida.                     |
| `404`  | Dominio no encontrado en el proyecto. |
| `500`  | Error al consultar el proveedor DNS.  |

## Reply domain (Reply-To limpio, sin token)

Configura un **dominio de reply branded** por proyecto (`reply.tudominio.com`) para que el Reply-To de los envíos API salga limpio —`localpart@reply.tudominio.com`— en vez del token codificado. El envío y la resolución de las respuestas son automáticos una vez verificado. Sin reply domain configurado, se usa el token de siempre. Cómo afecta el webhook: [Webhooks · reply domain](/api-reference/webhooks#reply-domain-branded-opcional-sin-token).

#### Request Body

| -------- | ------ | --------- | ------------------------------------------------------------------ |
| `domain` | string | Sí        | Dominio del cliente (ej. `phrasso.com`). RQE arma `reply.phrasso.com`. |

<Tabs><Tab title="cURL">

```bash
curl -X POST https://api.reallyquickemails.com/domains/reply \
```

</Tab></Tabs>


## Tracking domain (links de click con tu dominio)

Envuelve los links de click-tracking con **tu propio subdominio** (`links.tudominio.com`) en vez del compartido de RQE. Aísla la reputación del tracking y alinea los links con tu `From`. Se configura con un solo registro **CNAME**.

<Info>
El tracking domain **se cambia, nunca se borra**: los links de los emails ya enviados dependen de que el dominio siga resolviendo. Por eso no hay endpoint `DELETE` — para migrar, publica el CNAME nuevo y llama a `PUT` con el nuevo dominio.
</Info>

Estos endpoints cuelgan de `/api/projects/:projectId`. El `:projectId` es el **UUID de tu proyecto** —el `project_id` que devuelve cualquier respuesta de envío— y debe coincidir con el proyecto de tu API key.

### GET /api/projects/:projectId/tracking-domain

Estado actual del tracking domain del proyecto.

<Tabs><Tab title="cURL">

```bash
curl https://api.reallyquickemails.com/api/projects/:projectId/tracking-domain \
  -H "Authorization: Bearer sk_proj_xxxxxxxxxxxx"
```

</Tab></Tabs>

**Respuesta** `200 OK`

```json
{
  "domain": "links.tudominio.com",
  "status": "verified",
  "verified_at": "2026-07-20T12:00:00.000Z",
  "cname": { "name": "links.tudominio.com", "target": "track.reallyquickemails.com" }
}
```

`domain`, `status` y `cname` son `null` si no hay ninguno configurado.

### PUT /api/projects/:projectId/tracking-domain

Setea o cambia el tracking domain. Queda en `pending` hasta verificar. Debe ser un subdominio propio (ej. `links.tudominio.com`), nunca un dominio de RQE.

#### Request Body

| Campo    | Tipo   | Requerido | Descripción                                    |

```bash
curl -X PUT https://api.reallyquickemails.com/api/projects/:projectId/tracking-domain \

```

</Tab></Tabs>

**Respuesta** `200 OK`

```json
{
  "domain": "links.tudominio.com",

  "cname": { "name": "links.tudominio.com", "target": "track.reallyquickemails.com" },
```

Chequea el DNS y marca `verified` si el CNAME ya resuelve a nuestro target (soporta el CNAME flattening de Cloudflare comparando registros A).

## Verificacion de Email (Magic Link)

Cuando el estado cambia (de `pending` a `verified` o `failed`), RQE emite el webhook `sender.verified` o `sender.failed`. Ver [Webhooks](/api-reference/webhooks).
